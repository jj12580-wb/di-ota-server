# OTA 服务器技术方案

## 1. 文档信息
- 文档名称：OTA 服务器技术方案
- 版本：v1.0
- 日期：2026-04-27
- 关联文档：需求文档.md

## 2. 目标与约束

### 2.1 建设目标
1. 构建稳定、安全、可观测的 OTA 服务端。
2. 支持 10 万级设备灰度发布、暂停、恢复、回滚。
3. 控制升级风险，降低失败率，提升问题定位效率。

### 2.2 明确约束
1. 不需要审批流，发布工程师有权限即可发布。
2. 不要求断点续传，下载失败后仅支持整包重试。
3. CDN 为必选组件，设备下载必须走 CDN。
4. 系统自用，不做计费；保留用量统计能力。

## 3. 总体架构

### 3.1 架构分层
1. 接入层：
   - 管理端 API（包管理、任务管理、设备管理、审计查询）
   - 设备端 API（检查更新、状态上报）
2. 业务层：
   - 包管理服务
   - 版本策略服务
   - 发布任务服务
   - 升级编排与状态机服务
   - 统计与告警服务
3. 数据与基础设施层：
   - PostgreSQL（核心业务数据）
   - Redis（缓存、幂等键、限流计数）
   - RabbitMQ（异步事件流）
   - 对象存储（S3）
   - CDN（升级包分发）
4. 可观测层（轻量方案）：
   - Metrics（Prometheus + Grafana）
   - Logs（结构化日志 + Loki）
   - Trace（首期不建设分布式追踪）

### 3.2 后端技术栈
1. 编程语言与框架：Go 1.23 + Gin。
2. 数据访问：PostgreSQL + sqlc。
3. 缓存与队列：Redis + RabbitMQ。
4. 对象存储与 CDN：S3 协议兼容对象存储 + CDN。
5. 鉴权与安全：JWT、HMAC-SHA256、TLS。
6. 运维与发布：Docker + Docker Compose（必需）。

### 3.3 前端技术栈
1. 框架：React 18。
2. 工程化：Vite + TypeScript。
3. 组件库：Ant Design。
4. 状态管理：Zustand。
5. 接口调用：Axios。

### 3.4 架构说明
1. 设备检查更新接口与状态上报接口走无状态部署，可水平扩容。
2. 长耗时动作（统计聚合、告警计算、审计归档）异步化，保证接口低延迟。
3. 当前采用单 PostgreSQL 实例，缓存承担热点查询与限流压力。
4. 升级包分发链路为 对象存储 -> CDN -> 设备，源站不直接暴露给设备。

## 4. 模块设计

### 4.1 包管理模块
1. 上传升级包并校验：文件大小、SHA-256、签名。
2. 维护包状态：Draft、Published、Disabled、Archived。
3. 维护包元数据：
   - package_id、product_code、version
   - hardware_version_min/max
   - min_upgradable_version、depends_on_version
4. 生成设备下载链接：返回 CDN 短时效签名 URL。

### 4.2 版本策略模块
1. 支持语义化版本与自定义版本格式。
2. 支持版本比较规则配置与升级路径限制。
3. 支持强制升级与非强制升级策略。
4. 支持升级时间窗与黑名单时间段。

### 4.3 设备管理模块
1. 设备注册与鉴权。
2. 设备标签管理：地区、客户、批次、硬件版本等。
3. 支持动态设备分组和表达式筛选。
4. 支持设备维度查询：当前版本、最后心跳、升级历史。

### 4.4 发布任务模块
1. 创建任务：选择 package、目标设备、灰度策略、阈值策略。
2. 任务控制：start、pause、resume、terminate、rollback。
3. 灰度策略：按百分比、按设备列表、按区域/客户。
4. 熔断策略：
   - 失败率阈值触发自动暂停
   - 连续失败数阈值触发自动暂停

### 4.5 升级编排模块
1. check-update：判断设备是否命中当前可升级任务。
2. report-status：接收设备升级状态并驱动状态机。
3. 幂等保障：重复上报不重复计数、不破坏状态顺序。
4. 支持整包重试，不支持断点续传。

### 4.6 统计与告警模块
1. 实时聚合任务进度、成功率、失败率。
2. 聚合错误码分布、区域分布、型号分布。
3. 按规则触发告警：失败率异常、任务超时、下载异常激增。
4. 保留用量统计：下载次数、带宽、峰值并发。

### 4.7 审计模块
1. 记录关键操作：上传包、创建任务、暂停、恢复、回滚。
2. 审计日志支持检索与导出。
3. 日志不可篡改，保留操作前后快照。

## 5. 核心流程

### 5.1 包上传与发布
1. 发布工程师上传包。
2. 服务端完成哈希与签名校验后落对象存储，写入包元数据。
3. 包状态为 Draft，确认后发布为 Published。
4. 发布后可用于创建任务。

### 5.2 任务创建与启动
1. 选择已发布包与目标设备规则。
2. 设置灰度阶段、失败阈值、时间窗。
3. 写入任务并启动，状态 Draft -> Running。
4. 冻结目标设备快照，避免发布期间目标集合漂移。

### 5.3 设备检查更新
1. 设备携带鉴权信息调用 check-update。
2. 服务端校验设备身份、版本兼容、灰度命中、时间窗。
3. 命中时返回目标版本、SHA-256、签名、CDN URL。
4. 未命中时返回无可升级版本。

### 5.4 设备上报状态
1. 设备上报下载中/下载成功/校验失败/升级中/升级成功/升级失败/回滚结果。
2. 服务端执行状态转移校验与幂等去重。
3. 写升级记录并投递异步事件。
4. 异步更新统计、检查阈值、触发自动暂停，并按回滚规则执行回滚。

### 5.5 回滚流程
1. 支持人工触发与策略触发回滚。
2. 构建回滚任务，目标为已升级成功设备。
3. 状态 Running/Paused/Completed -> Rollbacking -> RolledBack。
4. 记录回滚原因、影响设备、完成率。

## 6. 数据库设计（核心表）

### 6.1 t_package
- 作用：升级包与元数据管理。
- 关键字段：package_id、product_code、version、file_hash、signature、status、publish_time。

### 6.2 t_device
- 作用：设备主数据与当前版本。
- 关键字段：device_id、product_code、hardware_version、current_version、last_heartbeat、tags。

### 6.3 t_release_task
- 作用：发布任务配置与状态。
- 关键字段：task_id、package_id、target_rule、canary_strategy、failure_threshold、state、schedule_time。

### 6.4 t_upgrade_record
- 作用：设备升级过程记录。
- 关键字段：device_id、task_id、source_version、target_version、state、error_code、state_time_*。
- 唯一约束：device_id + task_id。

### 6.5 t_audit_log
- 作用：关键操作审计。
- 关键字段：trace_id、operator、operation_type、resource_id、before_state、after_state、created_at。

### 6.6 t_task_stats
- 作用：任务统计快照。
- 关键字段：task_id、total_count、success_count、failed_count、failure_rate、error_distribution、snapshot_time。

## 7. 缓存与幂等设计

### 7.1 缓存键方案
1. package:{package_id}
2. device:{device_id}
3. task:{task_id}
4. task:running:list
5. stats:task:{task_id}

### 7.2 幂等策略
1. 管理端关键写操作支持 Idempotency-Key。
2. 设备上报使用 device_id + task_id + state + report_time 生成幂等键。
3. 幂等记录放 Redis，TTL 固定 24 小时。

## 8. 接口实现原则

### 8.1 管理端
1. 统一前缀：/api/v1
2. 关键接口：
   - POST /api/v1/packages
   - GET /api/v1/packages
   - POST /api/v1/release-tasks
   - POST /api/v1/release-tasks/{id}/pause
   - POST /api/v1/release-tasks/{id}/resume
   - POST /api/v1/release-tasks/{id}/rollback

### 8.2 设备端
1. 统一前缀：/device/v1
2. 关键接口：
   - POST /device/v1/check-update
   - POST /device/v1/report-status
   - GET /device/v1/packages/{packageId}/download

### 8.3 统一响应
1. 返回结构：code、message、data、trace_id、timestamp。
2. 业务成功 code=0；无可升级 code=2001。
3. 出错保留可定位错误码与错误详情。

## 9. 状态机设计

### 9.1 任务状态机
1. Draft -> Running -> Paused -> Running -> Completed
2. Running -> Failed
3. Running/Paused/Completed -> Rollbacking -> RolledBack

### 9.2 设备升级状态机
1. Pending -> Downloading -> Downloaded -> Verifying -> Upgrading -> Success
2. 任意阶段 -> Failed
3. Failed -> Rollbacking -> RolledBack / RollbackFailed

## 10. 安全方案

### 10.1 通信安全
1. 全链路 HTTPS/TLS。
2. 设备端请求签名校验与防重放校验。
3. 管理端采用 JWT。

### 10.2 访问控制
1. RBAC：管理员、发布工程师、只读、审计。
2. 关键操作二次确认。
3. 最小权限与操作审计闭环。
4. 管理端同时支持本地表单登录与 SSO 登录，SSO 通过配置开关启用。

### 10.3 升级包安全
1. 上传时校验哈希和签名。
2. 下发时返回哈希与签名，设备二次校验。
3. 禁止未签名包发布。

## 11. 可观测性方案

### 11.1 指标
1. API：QPS、延迟、错误率。
2. 任务：成功率、失败率、阶段耗时、覆盖率。
3. 分发：CDN 命中率、下载带宽、下载失败率。

### 11.2 日志
1. 管理操作日志。
2. 设备升级事件日志。
3. 结构化日志，统一 trace_id 贯穿，默认输出到 stdout 并采集到 Loki。

### 11.3 告警
1. 失败率超阈值。
2. 任务长时间无进展。
3. 下载异常激增。

## 12. 部署方案

### 12.1 环境规划
1. dev：开发联调。
2. staging：预发布验收与压测。
3. prod：生产高可用。

### 12.2 生产拓扑
1. API 服务多副本部署，支持水平扩容。
2. PostgreSQL 主从 + 自动故障转移。
3. Redis 哨兵模式。
4. RabbitMQ 集群高可用。
5. 对象存储 + CDN 必选。

### 12.3 容灾范围（当前阶段）
1. 当前阶段不建设异地容灾。
2. 保留本地定时备份与备份恢复演练。
3. 生产阶段再建设异地备份与跨可用区容灾。

### 12.4 Docker Compose 编排（必需）
1. 提供 compose 文件用于本地与测试环境一键启动。
2. 最小编排服务：ota-api、ota-worker、postgres、redis、rabbitmq。
3. 文件与配置：
   - docker-compose.yml：默认开发配置
   - docker-compose.staging.yml：预发覆盖配置
   - .env：端口、密钥、数据库连接、S3 配置
4. 启动顺序：postgres/redis/rabbitmq -> ota-worker -> ota-api。
5. 健康检查：api 健康端点、postgres 就绪探针、redis ping、rabbitmq 状态探针。

## 13. 容量与性能估算

### 13.1 初始目标
1. 日活检查设备：10 万台。
2. 峰值查询 QPS：2000+。
3. check-update 平均延迟 < 200ms，P95 < 500ms。

### 13.2 最简配置方案（首期）
1. OTA API：1 实例（2C4G）。
2. OTA Worker：1 实例（2C2G）。
3. PostgreSQL：1 实例（2C4G，单机）。
4. Redis：1 实例（1C2G，单机）。
5. RabbitMQ：1 实例（1C2G，单机）。
6. S3 + CDN：按量使用。
7. 该配置用于开发、联调和小规模试运行；进入生产高峰前升级为多副本高可用拓扑。

## 14. 测试与上线计划

### 14.1 测试策略
1. 单元测试：版本比较、状态机转移、阈值判定。
2. 集成测试：上传包 -> 发布 -> 升级 -> 回滚全链路。
3. 压力测试：check-update 与 report-status 双接口压测。
4. 故障测试：网络抖动、重复上报、组件故障切换。

### 14.2 里程碑
1. M1（2 周）：核心模型、设备接口、基础状态机。
2. M2（2 周）：包管理、任务管理、灰度与回滚。
3. M3（2 周）：统计告警、审计、权限系统。
4. M4（1-2 周）：压测、安全加固、试点上线。

## 15. 风险与应对
1. 设备端实现差异导致状态乱序。
   - 应对：协议版本化、严格状态转移校验、幂等去重。
2. 大规模下载带宽压力。
   - 应对：CDN 多地域分发、分批灰度、错峰策略。
3. 升级失败影响业务。
   - 应对：失败阈值自动熔断、快速回滚、人工预案。
