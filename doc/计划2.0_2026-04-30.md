# OTA 服务器改进计划 2.0

## 1. 文档信息
- 文档名称：OTA 服务器改进计划 2.0
- 版本：v2.0
- 日期：2026-04-30
- 基线文档：需求文档.md、技术方案.md
- 适用范围：当前代码实现基础上的下一阶段迭代规划

## 2. 计划目标

### 2.1 总体目标
1. 在现有可用闭环基础上，补齐核心缺口（用户管理、设备管理、策略、告警、权限）。
2. 将系统从“可用原型”推进到“可控上线”状态。
3. 形成可量化、可验收、可回滚的迭代节奏。

### 2.2 本轮原则
1. 先补“正确性与安全”再补“高级能力与体验”。
2. 页面优先落地，先完成管理台页面骨架、信息架构与交互，再由后端接口逐步对齐。
3. 每个阶段都要求：功能可测 + 指标可观测 + 文档可追溯。
4. 明确单租户自用定位，不引入多租户设计与实现。
5. 认证方案需要兼容 OAuth SSO 登录后获取上游 token，并由服务端代调用其他系统 API 的场景。

## 3. 当前实现评估（基于代码现状）

### 3.1 已具备能力
1. 基础包管理：上传 URL、上传完成、列表、详情、状态变更。
2. 基础任务管理：创建任务、任务状态控制、审计记录。
3. 设备协议基础：check-update、report-status、下载跳转。
4. 状态与安全基础：JWT、本地登录/OIDC、基础状态机校验、部分幂等处理。
5. Worker 基础统计：任务快照统计、阈值触发自动暂停。
6. 前端管理台：仪表盘、包管理、任务管理、详情页、响应式优化版本。
7. 本地环境：Docker Compose（dev/staging 覆盖）+ Keycloak + MinIO。

### 3.2 主要差距（对照需求与方案）
1. 用户管理缺失：缺少用户 CRUD、角色绑定、账号启停、密码重置。
2. 设备管理不足：缺少管理端设备查询/筛选/历史视图。
3. 版本策略不足：缺少语义化比较、升级路径限制、时间窗/黑名单。
4. 灰度策略不完整：缺少按设备列表、按区域/客户等多维策略。
5. 监控告警不足：缺少规则引擎、告警通知链路、异常分布统计落地。
6. 权限模型不足：缺少 RBAC 实体化、角色鉴权中间件与统一权限校验。
7. 审计能力不足：导出、按时间/操作人检索能力未完整落地。
8. 非功能能力不足：压测基线、限流、防重放、可观测看板仍缺。

## 4. 2.0 改进范围

### 4.1 必做（P0）
1. 用户管理 v1（用户列表/创建/禁用/重置密码/角色绑定）。
2. 设备管理闭环（管理端设备列表、详情、历史、过滤）。
3. 版本策略引擎 v1（版本比较、最小升级版本、升级路径限制）。
4. 发布策略 v1（时间窗、生效时间、黑名单时段）。
5. 告警能力 v1（失败率、任务卡住、下载异常三类规则）。
6. RBAC v1（管理员/发布工程师/只读/审计四角色）。
7. 集成测试补齐（上传 -> 发布 -> 上报 -> 暂停/回滚）。

### 4.2 应做（P1）
1. 审计导出（CSV/JSON）与组合检索。
2. 前端任务看板增强（异常分布、趋势图、任务健康度）。
3. 幂等能力强化（统一幂等键规范与重复请求回放策略）。
4. 性能基线建设（压测脚本 + 目标阈值）。

### 4.3 可选（P2）
1. 告警通知扩展（Webhook/企业 IM）。
2. 运营报表（日/周发布成功率、失败原因 TopN）。
3. 运维效率项（批量操作、批量导入导出增强）。

## 5. 分阶段实施计划

## 阶段 A：用户与权限基座
目标：先具备“谁能做什么”的可控能力，避免后续功能裸奔。

工作项：
1. 新增用户表、角色表、用户角色关联表及迁移脚本。
2. 管理端用户管理 API：列表、创建、启停、重置密码、角色绑定。
3. 后端统一鉴权中间件与路由权限声明。
4. 前端用户管理页面与基础角色控制（按钮级隐藏 + 接口级拦截）。

验收标准：
1. 可以完成“创建用户 -> 分配角色 -> 登录 -> 权限生效”闭环。
2. 未授权用户访问受限接口返回明确错误码。
3. 用户与权限相关测试覆盖关键路径。

## 阶段 B：策略与设备管理可用化
目标：让升级决策与设备查询达到可运维状态。

工作项：
1. 新增 `version_policy` 相关数据结构与查询。
2. 在 check-update 中接入版本比较与升级路径校验。
3. 接入任务时间窗、黑名单时间段判断。
4. 提供设备列表/详情/升级历史 API（支持分组、型号、状态筛选）。

验收标准：
1. 策略决策可解释（返回拒绝原因码）。
2. 管理台可完成“查设备 -> 看历史 -> 定位异常”。
3. 查询接口在 10w 设备数据下响应可接受（P95 < 800ms）。

## 阶段 C：告警与审计上线能力
目标：让系统具备“发现问题、追踪问题、回溯问题”的上线条件。

工作项：
1. 告警规则引擎 v1（失败率、无进展、下载异常）。
2. 告警事件存储与去重、抑制策略。
3. 审计日志支持时间范围、操作人、资源类型检索。
4. 审计导出接口（分页导出 + 异步任务可选）。

验收标准：
1. 告警规则可配置、可触发、可追踪。
2. 审计检索与导出可覆盖关键操作链路。
3. 关键链路有审计记录并可关联 trace。

## 阶段 D：性能与发布准备
目标：达到试点上线最低门槛。

工作项：
1. 压测场景：check-update/report-status 双接口。
2. 限流与防重放基线策略上线。
3. 失败场景演练（DB 抖动、MQ 不可用、存储延迟）。
4. 发布 Runbook、回滚预案、值班手册完善。

验收标准：
1. 核心接口达到既定延迟和错误率目标。
2. 演练通过并有记录。
3. 形成可执行上线清单。

## 6. 技术改进建议（架构与工程）

### 6.1 后端
1. 将策略决策集中为 `PolicyService`，避免规则散落在路由层。
2. 将任务控制动作抽象为状态机模块，保留迁移兼容层。
3. 用统一错误码与错误上下文结构，便于前后端对齐。
4. 将认证信息统一抽象为 `AuthContext`，同时容纳本地会话信息、SSO 用户信息、外部系统授权信息。
5. 对外部系统 API 调用采用服务端代理模式，不让前端直接持有并调用第三方 access token。

### 6.2 数据与异步
1. 将统计与告警计算解耦为事件驱动（优先 RabbitMQ）。
2. 明确 Redis 作用域：幂等键、热键缓存、限流计数。
3. 为高频查询补索引与慢查询观测（按阶段上线）。
4. 若 SSO 场景需要访问外部系统 API，需补充 token 元数据短时缓存与失效处理机制。

### 6.3 前端
1. 统一页面模式：列表页（筛选+表格+批量操作）/详情页（摘要+时间线）。
2. 增强可用性：空状态、错误态、加载态统一组件。
3. 新增用户管理页（用户列表、角色绑定、账号状态管理）。
4. 引入按角色的路由守卫与能力开关。
5. 认证状态拆分为“本系统登录态”和“外部系统授权态”，页面需明确展示二者区别。

### 6.4 认证与外部系统集成
1. 登录成功后的认证结果拆分为两层：
   - 平台自身会话：用于访问本系统 API。
   - 外部系统授权态：用于判断是否可以由服务端代调用外部系统 API。
2. 若 SSO 返回 access token，服务端需要区分两种模式：
   - 仅认证模式：只完成身份登录，不保留外部调用能力。
   - 认证 + 代调用模式：保留 token 元数据，通过后端代理访问外部系统。
3. 前端不直接调用第三方系统接口，统一通过本系统聚合接口访问，例如：
   - `GET /api/v1/integrations/users`
   - `GET /api/v1/integrations/devices`
4. 对外部系统调用需补充：token 失效处理、重新授权流程、错误码映射、审计记录。
5. 首期页面设计中，凡依赖外部系统数据的区域，都必须有“未授权/授权失效/调用失败”三种状态。

### 6.5 测试与质量
1. 增加 API 集成测试目录，构建最小端到端用例。
2. 增加契约测试，保证设备协议字段稳定。
3. 引入 CI 质量门：`go test`、前端构建、基础 lint。

## 7. 风险与应对
1. 策略复杂度上升导致回归风险。
   - 应对：策略引擎单元测试 + 回放样本测试。
2. 新增用户与权限可能影响现有联调。
   - 应对：灰度启用 RBAC，默认兼容旧行为。
3. 告警过多导致噪声。
   - 应对：去重、抑制窗口、分级告警。
4. 性能问题在晚期暴露。
   - 应对：阶段 C 前完成首次压测并优化一次。

功能：

### 11.4 DeviceDetailPage
目标：查看设备详情与升级历史。

## 12. 页面实施顺序建议
1. 先完成 MainLayout、LoginPage、DashboardPage 的页面骨架与认证状态表达，这是所有后续页面的统一入口。
2. 再补 UsersPage、DevicesPage、AlertsPage 的列表页骨架和占位交互，先把导航结构与页面信息架构补齐。
3. 然后增强 TasksPage/TaskDetailPage，因为这是发布控制核心页面。
4. 再增强 PackagesPage/PackageDetailPage，因为其依赖策略字段与状态机补齐。
5. 最后补全各页面与真实接口、外部系统授权态、权限裁剪的联调逻辑。
