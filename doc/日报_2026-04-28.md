# OTA Server 代码评审日报

**日期:** 2026-04-28
**评审范围:** 全量代码（Backend + Frontend + 基础设施配置）

---

## 一、项目概览

OTA Server 是一个设备固件远程升级平台，采用前后端分离架构：

- **Backend:** Go 1.22 + Gin + PostgreSQL + sqlc，分为 API 服务（`cmd/api`）和 Worker 服务（`cmd/worker`）
- **Frontend:** React 18 + TypeScript + Ant Design + Zustand
- **基础设施:** PostgreSQL 16, Redis 7, RabbitMQ 3, Keycloak 24, MinIO

核心业务流程：创建固件包 → 创建发布任务 → 设备轮询检查更新 → 上报升级状态。支持任务暂停/恢复/回滚和审计日志。

## 二、严重问题

### 1. 硬编码凭证登录（Critical）

**文件:** `backend/internal/server/router.go:66`

登录接口直接比对用户名密码：

```go
if req.Username != "admin" || req.Password != "Admin@123456" {
```

这是硬编码的超级管理员凭据。即使 .env 和 keycloak 配置中有独立的认证体系，这段代码仍然允许任何人用 `admin / Admin@123456` 绕过所有认证机制获取 JWT token。

**建议:** 生产环境应完全移除该登录方式，或至少从配置/数据库读取凭据并加盐哈希存储。

### 2. .env 文件提交到仓库（Critical）

**文件:** `.env`

项目根目录的 `.env` 包含了所有默认密钥（数据库密码、S3 密钥、JWT Secret、OIDC Client Secret），且与 `.env.example` 内容完全一致。虽然 `.gitignore` 已排除 `.env`，但两个文件内容相同意味着 example 文件本身也携带了弱密钥作为默认值。

**建议:** `.env.example` 中的敏感值应使用占位符（如 `<your-jwt-secret>`）而非实际弱值。

### 3. 密码以明文存储在 Keycloak 配置中（High）

**文件:** `deploy/keycloak/realm-ota-dev.json:44`

```json
"credentials": [{"type": "password", "value": "Admin@123456", "temporary": false}]
```

开发环境的初始用户密码明文存放在 realm JSON 导入文件中。

**建议:** 在 CLAUDE.md 或部署文档中明确标注此为 dev-only，禁止用于生产环境。

## 三、中等问题

### 4. 启动时自动执行 Schema（Medium）

**文件:** `backend/cmd/api/main.go:24`

```go
if err := db.ApplySchema(pg, "migrations/001_init.sql"); err != nil {
```

API 启动时会读取 SQL 文件并直接 `Exec`。这相当于在每个请求进程启动时跑 `CREATE TABLE IF NOT EXISTS`。问题在于：

- 无法处理后续迁移（只有 001，没有迁移工具）
- 多实例启动时存在并发执行 SQL 的风险
- 应该使用 Flyway、golang-migrate 等迁移工具管理版本

### 5. JWT 验证不完整（Medium）

**文件:** `backend/internal/server/router.go:454-455`

```go
parsed, err := jwt.Parse(tok, func(token *jwt.Token) (interface{}, error) {
    return []byte(cfg.Auth.JWTSecret), nil
})
```

未校验 `token.Method` 是否为预期的 HS256，攻击者可以改用 `none` 算法签名。

**建议:** 添加 `if token.MethodAlg() != jwt.SigningMethodHS256.Alg() { return nil, errors.New("invalid signing method") }`。

### 6. 状态管理无数据库约束（Medium）

**文件:** `backend/internal/server/router.go:24-35`, `backend/migrations/001_init.sql:19`

`t_release_task.state` 字段为 `VARCHAR(32)`，无 CHECK 约束。状态转换逻辑完全由应用层 `taskStateTransition` map 维护，数据库层不保证状态值的有效性。

**建议:** 添加 `CHECK (state IN ('Running', 'Paused', 'Completed', 'RolledBack', 'Failed'))`。

### 7. Worker 进程为空（Medium）

**文件:** `backend/cmd/worker/main.go:17-19`

```go
for {
    log.Printf("worker heartbeat")
    time.Sleep(30 * time.Second)
}
```

Worker 服务目前只打印心跳，未实现任何业务逻辑（如统计聚合、失败阈值自动回滚、RabbitMQ 消息消费等）。

### 8. Vite 缺少代理配置（Medium）

**文件:** `frontend/vite.config.ts`

前端调用 `/api/v1/ping` 但 Vite 配置未设置 `server.proxy`。在 Docker 环境中 `ota-console` 和 `ota-api` 是不同容器，前端直接请求 `/api/v1/*` 会打到 localhost:5173 而不是 8080。

**建议:** 添加 proxy 配置将 `/api` 转发到 `http://ota-api:8080`。

## 四、轻微问题

### 9. ID 生成基于时间戳（Low）

**文件:** `backend/internal/server/router.go:156, 204`

```go
id := fmt.Sprintf("pkg-%d", time.Now().UnixNano())
id := fmt.Sprintf("task-%d", time.Now().UnixNano())
```

同一纳秒内创建两个资源会产生 ID 冲突。单机低并发场景下概率极低，但不如使用 UUID。

### 10. 正则表达式重复编译（Low）

**文件:** `backend/internal/server/router.go:531-541`

`validatePasswordPolicy` 每次调用都重新 `regexp.MustCompile`。虽然 MustCompile 有内置缓存，但将正则编译为包级变量更清晰。

### 11. 断言错误未处理（Low）

**文件:** `backend/internal/server/router.go:466`

```go
sub, _ := claims["sub"].(string)
```

类型断言失败时静默忽略，`sub` 为空字符串时返回 `"unknown"`。应记录警告日志。

### 12. CreateIdempotency 未在调用方做预检查（Low）

**文件:** `backend/internal/server/router.go:420-424`

`report-status` 接口先写入升级记录再写入幂等表，但没有先查询幂等表判断是否已处理。ON CONFLICT 在 SQL 层解决了数据正确性，但应用层应优先检查以避免不必要的写入。

### 13. `t_task_stats` 表和 `t_device` 表未被使用（Low）

**文件:** `backend/migrations/001_init.sql:51-60, 22-29`

定义了 `t_task_stats` 和 `t_device` 表，但在 `queries/ota.sql` 中没有对应查询，Go 代码也未使用。属于未完成的功能骨架。

### 14. Redis 和 RabbitMQ 配置已加载但未使用（Low）

**文件:** `backend/internal/config/config.go`

Redis 和 RabbitMQ 的配置项已定义，但后端代码未引入 Redis 客户端或 AMQP 客户端。配置加载了但未被消费。

### 15. `.env` 与 `.env.example` 内容完全一致

两个文件应该有所区分：example 文件提供格式说明和占位符，实际 `.env` 携带真实值。当前两者内容相同。

## 五、代码结构评价

### 优点

- **sqlc 代码生成:** 使用 sqlc 从 SQL 文件生成类型安全的 Go 代码，查询和业务逻辑分离良好
- **统一响应格式:** 所有 API 返回 `{code, message, data}` 结构，前后端协议一致
- **审计日志:** 任务操作有完整的 before/after 审计记录，设计合理
- **设备端 API 设计:** `/device/v1/check-update` 和 `/report-status` 接口满足了设备轮询升级的核心需求
- **Docker Compose 编排:** 开发环境一键启动，健康检查齐全

### 改进方向

- 后端代码集中于 `router.go` 单文件（550 行），应按路由分组拆分为 handler 包
- 无单元测试、无集成测试
- 无请求速率限制，`/device/v1/check-update` 高频轮询可能打满数据库连接
- 无 graceful shutdown 处理

---

## 六、总结

当前代码为 OTA 平台 MVP 版本，核心流程（包管理 → 任务发布 → 设备检查 → 状态上报 → 审计）已打通。主要风险集中在安全方面（硬编码凭据、JWT 算法验证缺失），建议在接入生产环境前优先修复 Critical 和 High 级别问题。Worker 服务、任务统计、设备管理等模块仍处于骨架阶段，需继续迭代。
