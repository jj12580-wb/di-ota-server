# OTA 服务器需求文档

## 1. 文档信息
- 文档名称：OTA 服务器需求文档
- 版本：v1.0
- 日期：2026-04-27
- 状态：草案
- 适用范围：设备固件/软件 OTA 升级服务端

## 2. 项目背景与目标

### 2.1 背景
随着设备规模增长，人工升级方式效率低、风险高，且无法满足批量管理、分阶段发布、失败回滚和合规审计等要求，需要建设统一 OTA 服务器端能力。

### 2.2 项目目标
1. 提供稳定、安全、可观测的 OTA 升级服务端。
2. 支持大规模设备分批、灰度、定向升级。
3. 降低升级失败率并提供可追踪的回滚策略。
4. 满足企业级权限、安全、审计和运维需求。

### 2.3 非目标（当前阶段不包含）
1. 设备端 Bootloader 开发。
2. 固件编译与打包流水线改造。
3. 客户端 App 端升级 UI 设计。

## 3. 术语定义
- OTA（Over-The-Air）：通过网络远程下发升级包并执行升级。
- 升级包：固件或软件二进制文件及其元数据。
- 设备组：按产品线、区域、客户、硬件版本等维度划分的设备集合。
- 灰度发布：按比例、批次或标签逐步扩大升级范围。
- 回滚：升级失败或异常时恢复到上一稳定版本。

## 4. 角色与使用者
1. 平台管理员：维护系统配置、权限、发布策略。
2. 发布工程师：上传升级包、创建发布任务、监控发布进度。
3. 运维/支持人员：处理异常设备、查看日志和告警。
4. 审计人员：查看操作记录与发布记录。
5. 设备客户端：轮询升级策略、下载包、上报状态。

## 5. 业务流程（高层）
1. 上传升级包并校验完整性与签名。
2. 创建发布任务（指定目标设备、时间窗、灰度策略、约束条件）。
3. 开始发布任务。
4. 设备按策略检查升级、下载包、执行升级并上报状态。
5. 服务端实时统计成功率、失败率、耗时、异常类型。
6. 触发阈值后自动暂停/回滚，或人工干预。
7. 任务结束并生成审计报告。

## 6. 功能需求

### 6.1 包管理
1. 支持上传升级包（固件、补丁包、资源包）。
2. 记录包元数据：
   - 包 ID、名称、版本号、适配产品型号。
   - 兼容硬件版本范围、依赖版本、最小可升级版本。
   - 文件大小、哈希（SHA-256）、签名信息、发布时间。
3. 支持包状态：草稿、已发布、已禁用、已归档。
4. 支持包下载地址生成（短时效签名 URL）。
5. 支持包完整性校验失败的拦截。

### 6.2 版本与策略管理
1. 支持语义化版本（如 1.2.3）与自定义版本规则（如 YYYY.MM.DD、V2026R1），并可配置版本比较规则。
2. 支持升级路径限制（如仅允许跨 2 个小版本以内）。
3. 支持强制升级/可选升级策略。
4. 支持升级时间窗（例如 00:00-06:00）和黑名单时间段。

### 6.3 设备管理
1. 接入设备注册与鉴权。
2. 支持设备标签：产品、地区、客户、批次、硬件版本、在线状态等。
3. 支持设备分组和动态筛选。
4. 支持单设备查询：当前版本、最后心跳、升级历史、失败原因。

### 6.4 发布任务管理
1. 创建发布任务并选择目标设备（组、标签、条件表达式）。
2. 灰度能力：
   - 按比例（1%、5%、20%、100%）分批。
   - 按设备列表分批。
   - 按区域/客户定向发布。
3. 任务控制：开始、暂停、恢复、终止、回滚。
4. 失败阈值控制：
   - 失败率超过阈值自动暂停。
   - 连续失败台数超过阈值自动暂停。
5. 支持定时发布。

### 6.5 设备升级协议
1. 设备查询升级接口：返回是否可升级、目标版本、下载地址、校验信息。
2. 设备状态上报接口：下载中、下载成功、校验失败、升级中、升级成功、升级失败、回滚成功/失败。
3. 幂等要求：重复上报不会造成状态错乱。
4. 支持下载失败后的整体重试（不要求断点续传）。

### 6.6 监控与告警
1. 实时看板：任务进度、成功率、失败率、在线设备覆盖率。
2. 异常分布：按错误码、设备型号、区域聚合。
3. 告警规则：失败率阈值、任务超时、下载异常激增。

### 6.7 日志与审计
1. 记录关键操作日志：上传包、创建/变更任务、暂停/恢复/回滚。
2. 记录设备升级事件日志（可检索）。
3. 审计日志不可篡改，支持导出。
4. 支持按时间范围、操作人、任务 ID 检索。

### 6.8 权限与组织管理
1. 支持 RBAC（管理员、发布工程师、只读、审计）。
2. 支持多租户隔离（如面向不同客户/事业部）。
3. 关键操作支持二次确认。

## 7. 非功能需求

### 7.1 性能
1. 设备查询接口平均响应时间 < 200ms（P95 < 500ms）。
2. 支持至少 10 万台设备每日升级检查。
3. 峰值并发查询 QPS 依据压测目标设定（建议 2000+）。

### 7.2 可用性
1. 服务可用性目标：99.9%（阶段目标），逐步提升至 99.95%。
2. 关键组件高可用部署（无单点故障）。
3. 支持故障自动恢复与任务断点续执行。

### 7.3 安全
1. 全链路 HTTPS/TLS。
2. 设备鉴权（Token/证书）与请求签名校验。
3. 升级包必须签名，设备侧校验签名与哈希。
4. 最小权限原则与敏感操作审计。
5. 防重放、防刷接口、速率限制。

### 7.4 可观测性
1. 指标：请求量、错误率、延迟、任务成功率、下载带宽。
2. 日志：结构化日志，支持 traceId 关联。
3. 链路追踪：核心接口全链路可追踪。

### 7.5 兼容性
1. 兼容多产品线与多硬件版本。
2. 协议版本可演进，保持向后兼容。

## 8. 接口需求（草案）

### 8.1 管理端 API
1. `POST /api/v1/packages` 上传升级包。
2. `GET /api/v1/packages` 查询包列表。
3. `POST /api/v1/release-tasks` 创建发布任务。
4. `POST /api/v1/release-tasks/{id}/pause` 暂停任务。
5. `POST /api/v1/release-tasks/{id}/resume` 恢复任务。
6. `POST /api/v1/release-tasks/{id}/rollback` 回滚任务。

### 8.2 设备端 API
1. `POST /device/v1/check-update` 检查升级。
2. `POST /device/v1/report-status` 上报升级状态。
3. `GET /device/v1/packages/{packageId}/download` 获取下载地址或重定向。

### 8.3 返回码规范（示例）
- `0`：成功
- `1001`：鉴权失败
- `1002`：参数非法
- `2001`：无可用升级
- `3001`：包校验失败
- `5000`：系统内部错误

## 9. 数据模型（核心实体）
1. Device：设备基础信息、标签、当前版本、状态。
2. Package：升级包元数据、哈希、签名、存储地址。
3. ReleaseTask：任务配置、灰度策略、状态机。
4. UpgradeRecord：每台设备每次升级记录。
5. AuditLog：关键操作审计记录。

## 10. 状态机要求

### 10.1 发布任务状态
- Draft -> Running -> Paused -> Running -> Completed
- Running -> Failed
- Running/Paused -> Rollbacking -> RolledBack

### 10.2 设备升级状态
- Pending -> Downloading -> Downloaded -> Verifying -> Upgrading -> Success
- 任意阶段 -> Failed
- Failed -> Rollbacking -> RolledBack / RollbackFailed

## 11. 部署与运维需求
1. 支持容器化部署（Docker/Kubernetes）。
2. 配置中心与密钥管理（如 Vault/KMS）。
3. 对象存储用于升级包分发（S3/OSS/MinIO）。
4. CDN 为必选项，用于提升大规模下载能力与跨地域分发稳定性。
5. 支持多环境：dev/staging/prod。

## 12. 测试验收标准

### 12.1 功能验收
1. 可完成端到端升级闭环（上传包 -> 发布 -> 设备升级 -> 统计）。
2. 灰度发布与暂停/恢复/回滚功能可用。
3. 设备重复上报、断线重连场景处理正确。

### 12.2 性能验收
1. 达到既定 QPS 和响应延迟目标。
2. 大批量设备并发升级时系统稳定，无明显错误峰值。

### 12.3 安全验收
1. 未签名包无法发布。
2. 未授权设备无法获取升级包。
3. 审计日志可追溯关键操作。

## 13. 里程碑建议
1. M1（2 周）：核心数据模型、设备检查与状态上报 API。
2. M2（2 周）：包管理、发布任务、灰度与任务控制。
3. M3（2 周）：监控告警、审计、权限模型。
4. M4（1-2 周）：压测、安全加固、试点上线。

## 14. 风险与应对
1. 风险：设备端实现不一致导致状态异常。
   - 应对：定义严格协议版本与兼容策略，提供设备端 SDK 示例。
2. 风险：大包下载导致带宽瓶颈。
   - 应对：对象存储 + CDN + 分区域分发。
3. 风险：升级失败影响业务。
   - 应对：灰度阈值自动熔断 + 回滚机制 + 人工干预预案。

## 15. 待确认事项
1. 目标设备规模（当前/未来 12 个月）。
2. 是否要求多租户隔离（当前为自用场景，不涉及计费，可保留用量统计）。
3. 升级窗口与业务高峰期限制。
4. 是否必须支持差分包升级。
